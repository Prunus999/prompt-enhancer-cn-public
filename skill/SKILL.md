# Prompt Enhancer CN

把用户粗糙的自然语言意图，改写成更强、更清晰、可直接使用的高质量 Prompt。默认强调实用改写，而不是讲 Prompt 理论。

## 快速使用

1. 识别目标模型或工具：
   - 如果用户明确说了模型、平台或工具，就针对该目标优化。
   - 如果没说，就给出通用版本，并简要标注假设。
2. 判断任务类型：
   - 编码 / 调试
   - 写作 / 总结
   - 调研 / 分析
   - 图像生成
   - Agent / 工作流指令
3. 提取关键缺失信息：
   - 目标
   - 输入材料
   - 约束
   - 输出格式
   - 质量要求
4. 默认输出中文优化稿，除非用户明确要求其他语言。
5. 优先返回可直接使用的 Prompt，解释说明放后面。

## 工作方式

- 优先直接优化，不做冗长教学。
- 如果缺少重要信息但仍可继续，就明确写出假设，不要轻易卡住。
- 只有在目标模型、交付物、风险边界确实无法判断时，才追问。
- 除非任务明显需要详细规范，否则保持 Prompt 紧凑。
- 保留用户原有领域语言，重点提升结构、清晰度和约束表达。

## 输出约定

默认输出结构：

1. `优化后的 Prompt`
2. `可选补充`

`可选补充` 仅在有价值时提供：
- 假设说明
- 简短使用说明
- 一个或两个变体，如 `简洁版` / `严格版`

如果用户没有指定输出形式，默认采用以下模式之一：
- `只输出优化稿`：只给最终 Prompt，适合直接复制使用。
- `优化稿 + 说明`：给出最终 Prompt，并简要说明关键改动点。

如果用户明确说“不要解释”“直接给 prompt”，则强制使用 `只输出优化稿`。

## 改写规则

- 只有在角色设定确实能提升结果时，才加入角色。
- 先写任务，再写背景，不让背景淹没目标。
- 把模糊描述改成可观察、可判断的要求。
- 缺少输出格式时，主动补上。
- 把硬约束和偏好分开。
- 对多步骤任务，可以要求分阶段处理，但最终只输出用户真正需要的内容。
- 不要编造事实、虚构限制或加入无意义的仪式化措辞。
- 除非任务本身要求展开，否则避免“尽可能详细”这类空泛指令。

## 按任务类型处理

### Coding / debugging

- 明确仓库上下文、约束、验收标准和验证方式。
- 如果用户要的是落地结果，优先给补丁式、实现式输出，而不是泛泛建议。
- 常用结构：上下文、目标、约束、交付物、验证。
- 如果涉及代码生成，要补上语言、框架、文件范围和边界情况。

需要模板时，读取 `references/templates.md` 对应章节。

### Writing / summarization

- 明确受众、语气、长度、材料边界和禁止事项。
- 把期望结构写清楚，例如标题、要点、表格、摘要等。

需要模板时，读取 `references/templates.md` 对应章节。

### Research / analysis

- 当用户在意准确性时，要求结论尽量对应证据。
- 区分事实、假设、推断、建议。
- 明确比较维度、时间范围和输出格式。
- 如果用户需要引用、来源、可核查性，主动补充来源要求，例如：
  - 给出引用链接或来源名称
  - 标注“事实 / 推断 / 建议”
  - 证据不足时明确说明不足之处

需要模板时，读取 `references/templates.md` 对应章节。

### Image generation

- 强调主体、构图、风格、光线、镜头、材质、色板和排除项。
- Prompt 要保持视觉化，不要把非视觉性的业务要求塞进图像提示词。
- 默认可输出英文图像 Prompt；如果用户明确要求中文或目标模型更适合中文，再输出中文版本。

需要模板时，读取 `references/templates.md` 对应章节。

### Agent / workflow prompts

- 明确自治程度、工具边界、阶段汇报点和停止条件。
- 优先使用可执行、可检查的指令，不用激励式空话。

需要模板时，读取 `references/templates.md` 对应章节。

## 常见转换策略

- 如果输入只是一个一句话想法，就扩展成可直接使用的 Prompt。
- 如果输入过于啰嗦，就压缩成更清晰的版本。
- 如果输入把目标和做法混在一起，就拆开处理。
- 如果用户说“更专业”，通常要补的是约束、示例、输出格式，而不是堆修饰词。
- 如果用户说“更适合 AI”，就把任务拆解方式和输出契约写清楚。

## 响应形态示例

推荐形态：

- 用代码块直接给最终 Prompt。
- 如有必要，再补一小段使用说明。

不推荐形态：

- 在最终 Prompt 前先讲一大段 Prompt 工程理论。
- 明明一个强版本就够，却给出一堆互相竞争的框架。
